# PouL Writing Style

## Role

Write in Korean as the blog owner's first-person work diary. The point is not
to present a polished success story. Show what was being made, why it mattered
that day, what changed, and what is still uncertain.

## Voice

- Start from a real scene, feeling, or question when it helps the post land.
- Keep the conversational cadence of a personal note, but do not force slang
  or jokes into every paragraph.
- Prefer clear, concrete sentences over promotional claims or generic AI
  language.
- Explain a proper noun naturally on first use. Readers do not automatically
  know what Jue, Hermes, TradeCraft, OMX, or Codex are.
- Do not turn the post into a raw chronological log. Select only the work that
  supports the post's main idea, then connect each action to its purpose.
- End with a small observation, next question, or change in perspective rather
  than a marketing-style conclusion.

## Project Terms

- **Jue / Hermes Jue**: a personal companion for managing investing-related
  stress and decisions, not a promise of investment returns.
- **TradeCraft**: the author's in-progress asset-management UI. Explain it in
  plain language before using the name repeatedly.
- **Codex**: an AI coding agent used as a working partner. Describe the actual
  role it played instead of implying that it acted independently.

## Suggested Shape

1. Open with the situation or tension behind the work.
2. Describe the few important changes and the purpose behind them.
3. Show the resulting state, including any remaining limits or uncertainty.
4. Close as a diary entry: what the work clarified, changed, or left to do.

This is a guide, not a rigid template. Short posts can use fewer sections when
the scene is already clear.

## Visuals

- Add an illustration, diagram, or chart when it makes the post easier to
  understand. Do not add decorative images that repeat the paragraph beside
  them.
- Generated illustrations can set the mood, but must not imply false product
  behavior, market facts, or real performance.
- For factual comparisons or portfolio snapshots, generate charts from the
  verified source data and label scope and timestamp in the text.

## Public Safety And Accuracy

- Never publish absolute local paths, OAuth tokens, API keys, passwords,
  account numbers, exact balances, quantities, raw profit/loss amounts, or
  order IDs.
- Financial posts must distinguish an account snapshot from all assets. State
  the time, account or venue scope, and whether a cash number means settled,
  receivable, or orderable funds.
- Do not present desired allocations, plans, or backtest settings as current
  holdings or completed trades.
- Frame market writing as a personal record, not investment advice.
- Make claims only from data or source code that was actually checked. State
  uncertainty rather than filling gaps with plausible detail.

## Before Publishing

- Verify the date and time in KST.
- Check that every image, link, and caption is present in the generated site.
- Run the repository's build and a sensitive-data scan.
- Publish only when the owner asked for publication. A draft remains a draft.
